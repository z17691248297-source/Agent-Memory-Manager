# 国内大模型推理 / Agent 系统岗位面试准备

> 更新时间：2026-06-09  
> 适用岗位：大模型推理工程师、LLM Infra 工程师、推理优化工程师、Agent 系统工程师、AI 平台工程师  
> 说明：本文基于公开岗位 JD、公开面经和 `vLLM` 官方资料整理。官方招聘页优先，职位聚合页和面经页主要用于观察行业常见要求与高频问法。

## 1. 这些岗位到底在招什么

如果把国内相关岗位按能力模型拆开，通常不是在招“会调 API 的人”，而是在招下面这几类能力的组合：

1. **模型推理与部署**
   能把开源模型稳定跑起来，理解 `prefill`、`decode`、`KV Cache`、batching、量化、并行策略，并能分析吞吐、时延和显存占用。
2. **系统与工程能力**
   熟悉 Linux、Python/C++、CUDA/Triton、Docker/K8s、监控、Profiling、线上服务稳定性。
3. **大模型基础知识**
   理解 Transformer、Attention、RoPE、GQA/MQA、MoE、量化、并行训练/推理的基本原理。
4. **Agent / RAG / 工具调用**
   理解多轮上下文、工具调用、长期记忆、评测、提示词管理，以及如何避免上下文和缓存膨胀。
5. **性能优化与系统设计**
   能把问题拆成“瓶颈在哪里、怎么测、怎么改、怎么证明改得对”。

## 2. 从公开 JD 提炼出的高频要求

结合上海 AI Lab、字节等官方岗位页，以及聚合岗位页中和大模型推理/Agent 高度相关的描述，可以归纳出下面这些高频要求。

### 2.1 基础栈

- 熟悉 `Python`，很多岗位还要求 `C/C++`
- 熟悉 `Linux`、多进程、多线程、网络与系统调试
- 有 `CUDA`、`Triton`、`TensorRT-LLM`、`vLLM`、`SGLang`、`llama.cpp` 等经验会明显加分
- 了解 `PyTorch`、Hugging Face、分布式推理与 GPU 调优

### 2.2 推理优化方向

- `KV Cache` 管理与复用
- `continuous batching`
- `prefix caching`
- 长上下文优化
- 推理并行：`TP/PP/DP/EP`
- 量化：`FP8/INT8/INT4/AWQ/GPTQ`
- 推测解码 `speculative decoding`
- `PD 分离`（Prefill-Decode disaggregation）
- 显存碎片、显存峰值控制和吞吐/时延平衡

### 2.3 Agent 方向

- 多轮对话与工具调用流程设计
- 长生命周期任务的上下文管理
- 提示词模板、工具描述和历史消息压缩
- 结构化工具返回值、按需加载和外部存储
- 评测：成功率、时延、成本、稳定性

### 2.4 你这个比赛与岗位的直接重合点

这道赛题和真实岗位的重合度非常高，尤其对应下面几类能力：

- 长上下文和 `KV Cache` 管理
- 智能体多阶段推理中的内存生命周期管理
- 工具调用结果不直接灌满上下文，而是做结构化存储和摘要
- 前缀共享、分支共享、冷热分层存储
- Benchmark 设计和性能归因

换句话说，只要项目做扎实，这个比赛完全可以包装成一段很像真实业务的 `LLM Infra / Agent System` 项目经历。

## 3. 从公开面经看，面试一般怎么问

公开面经里，这类岗位很少只问八股，通常是下面这个节奏：

1. **先问项目**
   让你讲一个你做过的系统，重点追问你改了什么、为什么这么改、怎么证明有效。
2. **再问原理**
   典型会追问 `Transformer`、`Attention`、`KV Cache`、`vLLM`、量化、并行、MoE。
3. **然后问系统设计**
   比如“如何设计一个高吞吐推理服务”“如何优化长上下文”“如何支持多租户和分支推理”。
4. **最后看工程能力**
   会问如何做 profiling、如何定位瓶颈、如何做指标监控、如何压测、如何回归验证。

公开面经中已经能看到一些很直接的关键词，例如：

- `VLLM 原理`
- `continuous batching`
- `TP 矩阵如何切分`
- `MoE`
- `RL pipeline`
- “应用岗更关注项目和系统设计，基础八股相对少一些”

因此，准备策略不能只是背定义，而要准备“问题背景 -> 原理 -> 取舍 -> 实现 -> 指标”的完整答案链。

## 4. 高频原理解释：面试时应该怎么答

下面这些内容建议你反复讲熟。真正面试时，不需要一字不差照背，但逻辑框架最好稳定。

### 4.1 什么是模型权重、推理引擎、API 服务，它们之间是什么关系

一个清晰说法是：

- **模型权重**：模型本身的参数文件，例如 `Qwen2.5-7B-Instruct`
- **推理**：把输入 prompt 变成输出 token 的计算过程
- **推理引擎**：负责高效执行推理的软件系统，例如 `vLLM`
- **API 服务**：把推理能力封装成可调用接口，例如 OpenAI 兼容接口

最容易讲清楚的一句话是：

> 云 API 是调用别人部署好的模型；`vLLM` 是把你自己的开源模型部署成高性能推理服务。

### 4.2 一次推理请求内部发生了什么

可以按下面顺序回答：

1. 用户输入文本先经过 tokenizer 变成 token ids
2. 模型执行 **prefill**：把已有 prompt 整体读入，计算每层的中间状态
3. 这些中间状态里，和自注意力复用相关的部分会保存成 **KV Cache**
4. 进入 **decode** 阶段：每次只生成一个或少量 token
5. 生成新 token 时，不需要重算前面所有 token 的注意力，只需要复用 KV Cache，再补算新增 token
6. 随着上下文增长，KV Cache 也会增长

面试时最好补一句：

> `prefill` 更偏计算密集，`decode` 更偏访存密集，这是很多推理优化和 PD 分离的基础。

### 4.3 什么是 KV Cache，为什么它这么重要

`KV Cache` 是 Transformer 在推理时缓存的 `Key` 和 `Value`。  
作用是避免模型在生成下一个 token 时重复计算前面所有 token 的注意力相关中间结果。

它重要，是因为它直接决定了：

- 长上下文时能不能接受的时延
- 并发场景下的显存压力
- 是否能高效支持多轮对话和智能体长生命周期任务

一个常用近似公式可以这样说：

```text
KV Cache bytes
≈ batch_size × seq_len × num_layers × 2 × num_kv_heads × head_dim × dtype_bytes
```

说明：

- `2` 是因为有 K 和 V
- 如果模型使用 `GQA/MQA`，这里用的是 `num_kv_heads`，而不是全部 attention heads
- `seq_len` 增长时，KV Cache 近似线性增长

面试时常见追问：

- 为什么 `GQA` 能省 KV？
  因为多个 query heads 共享更少的 KV heads，所以同样上下文长度下缓存规模更小。
- 模型权重和 KV Cache 哪个是固定的？
  权重通常固定；KV Cache 是请求运行时增长的动态内存。

### 4.4 为什么智能体场景比普通对话更容易“爆显存”

普通对话大多是单线增长；智能体场景会叠加更多问题：

1. **生命周期长**
   一个任务可能要多轮规划、执行、反思。
2. **重复前缀多**
   system prompt、tool schema、规则说明经常反复出现。
3. **会分支**
   例如同时尝试方案 A/B/C，前缀相同但尾部不同。
4. **工具输出大**
   网页文本、JSON、检索结果、数据库结果都可能很长。

所以智能体场景的难点不只是“上下文长”，而是：

> 长、重复、分支、工具数据大，这几个问题同时出现。

这正是比赛题目和真实 Agent 岗位的共同核心。

### 4.5 vLLM 为什么快，它解决了什么问题

一个稳妥答法是：

`vLLM` 的核心价值不是“能跑模型”，而是“能更高效地跑很多请求”，尤其在长上下文和高并发下。

它主要解决几类问题：

- `KV Cache` 管理效率低
- 动态请求下 batch 利用率低
- 显存碎片严重
- 长 prompt 和多轮场景吞吐不足

面试时建议提三个关键词：

1. **PagedAttention**
2. **continuous batching**
3. **prefix caching**

如果时间允许，再补充 `speculative decoding`、chunked prefill、KV offload。

### 4.6 什么是 PagedAttention，为什么它是 vLLM 的代表性设计

这题几乎是 `vLLM` 面试必考。

传统做法常常要求一段序列的 KV Cache 连续存放，但在线服务里请求长度不一样、结束时间不一样，很容易造成：

- 显存碎片
- 大量预留空间浪费
- 迁移和扩容困难

`PagedAttention` 的核心思想类似操作系统里的分页：

- 把 KV Cache 按固定大小切成 block/page
- 逻辑上一个序列还是连续的
- 物理上这些 block 不需要连续存放
- 通过 block table 做映射

好处是：

1. 显存利用率更高
2. 减少外部碎片
3. 更容易做共享和回收
4. 更适合动态批处理和在线服务

你可以用一句更工程化的话总结：

> PagedAttention 让“序列连续”变成逻辑概念，而不是物理分配约束，从而把 KV Cache 管理从大块连续内存分配，变成了更灵活的块管理问题。

### 4.7 什么是 continuous batching，为什么它对吞吐很重要

静态 batching 是先凑一批请求，再统一执行；问题是不同请求长度不同，短请求会等长请求，GPU 利用率也会下降。

`continuous batching` 的思想是：

- 不等整批完全结束
- 在 decode 迭代的边界上，动态把新请求加入调度
- 让 GPU 尽量持续有活干

优点：

- 更高吞吐
- 更好的设备利用率
- 在在线服务里更实用

代价：

- 调度器更复杂
- 公平性更难保证
- KV Cache 管理更复杂
- 更容易暴露显存碎片和长尾问题

面试时可以补一句：

> continuous batching 本质是以更复杂的调度换取更高的整体吞吐和更低的平均资源浪费。

### 4.8 什么是 prefix caching，适合什么场景

如果很多请求前缀相同，例如：

- 同一个 system prompt
- 同一套工具说明
- 同一份长文档前缀

那么这些公共前缀的 KV 没必要每次都重新算。

`prefix caching` 的思路是：

- 识别相同前缀
- 复用已经算好的前缀 KV
- 后续只对新增尾部做 prefill/decode

适合的场景：

- 多轮会话
- 多用户共享系统提示词
- RAG/Agent 中工具定义固定、用户输入变化较小的场景

和你比赛题的关系非常直接：  
system prompt、tool schema、规则提示都可以成为前缀复用对象。

### 4.9 什么是 chunked prefill，为什么长 prompt 时会提到它

长 prompt 的 prefill 会占用很长时间，还可能阻塞 decode 请求。  
`chunked prefill` 的思路是把一个很长的 prefill 拆成多个 chunk，与其他请求更灵活地交错执行。

好处：

- 改善长 prompt 对队列的阻塞
- 改善 tail latency
- 让调度更平滑

代价：

- 调度逻辑更复杂
- 中间状态管理更细

面试时如果被问“长上下文怎么优化”，除了模型结构和 RoPE 扩展，服务侧就可以提这类技术。

### 4.10 为什么会有 PD 分离（Prefill / Decode Disaggregation）

`prefill` 和 `decode` 的硬件瓶颈不一样：

- `prefill` 更偏算力密集
- `decode` 更偏内存带宽和缓存访问

因此可以把两者拆到不同资源池：

- 一类设备专门做大吞吐 prefill
- 一类设备专门做稳定 decode

这样做的目标通常是：

- 提升整体设备利用率
- 降低长 prompt 对 decode 请求的干扰
- 让服务更容易按流量特征扩容

缺点是：

- KV 需要在 prefill 节点和 decode 节点之间转移
- 网络和同步开销会上升
- 架构复杂度和故障面变大

如果面试官追问“什么时候不适合 PD 分离”，你可以答：

- 请求规模不大时，转移开销可能不值
- 网络带宽不足时，KV 迁移会成为新瓶颈
- 小模型或低并发时，单机往往更简单高效

### 4.11 量化到底在优化什么，它和 KV Cache 是一回事吗

不是一回事。

- **权重量化**：压缩模型参数本身，主要影响模型常驻显存和算子执行方式
- **KV Cache 优化/量化**：压缩运行时缓存，主要影响长上下文和并发场景

面试时可以这样区分：

1. 权重量化主要解决“模型装不装得下、算得快不快”
2. KV Cache 优化主要解决“请求跑起来后，长上下文和多请求会不会爆显存”

常见权重量化：

- `INT8`
- `INT4`
- `AWQ`
- `GPTQ`
- `FP8`

常见代价：

- 精度损失风险
- 算子支持受限
- 部分场景吞吐提升不稳定

### 4.12 什么是 speculative decoding，它为什么可能提速

思路是先让一个更小、更快的 draft model 或草稿机制先猜一串 token，再让主模型验证。

如果猜对很多：

- 主模型就不用一个 token 一个 token 地严格生成
- 可以减少部分串行生成开销

什么时候收益更明显：

- 主模型很大
- 生成阶段是瓶颈
- draft 成本低且命中率高

什么时候收益一般：

- draft 模型不够准
- 验证成本高
- 系统瓶颈不在 decode

面试时的关键不是背定义，而是知道它是“用便宜猜测换贵模型串行步骤减少”的方法。

### 4.13 为什么说智能体内存管理不只是 KV Cache 问题

这是你比赛项目非常值得讲的一点。

智能体系统里的内存至少有三层：

1. **模型内部运行时内存**
   例如 KV Cache
2. **上下文工作内存**
   包括消息历史、system prompt、工具定义、摘要
3. **外部工具与环境状态**
   例如工具返回 JSON、检索结果、网页内容、数据库结果

如果只盯着 KV，而不控制后两层，效果往往有限。  
因为很多“显存爆炸”的根源，是你把大量不该进上下文的外部数据直接塞进了 prompt。

### 4.14 如果让你设计一个面向 Agent 的内存管理系统，你怎么答

面试时可以按“分层 + 生命周期 + 策略”来答。

#### 一层：对象分类

把智能体运行中的信息分成几类：

- 固定前缀：system prompt、tool schema、角色描述
- 会话历史：用户消息、模型回复、阶段摘要
- 分支状态：计划树、候选解、回滚点
- 工具数据：检索结果、网页文本、结构化 JSON
- 运行时缓存：KV Cache

#### 二层：生命周期

- 固定前缀：高复用、长生命周期，适合共享
- 会话历史：中生命周期，适合摘要和压缩
- 工具数据：很多是冷数据，适合外置存储
- KV Cache：热数据，优先留在 GPU，必要时分层迁移

#### 三层：核心策略

1. **前缀去重与共享**
   system prompt 和工具定义不重复注入
2. **历史摘要**
   老对话做摘要，只保留最近窗口 + 长期记忆摘要
3. **工具结果外置**
   原始结果放对象存储或本地结构化存储，只把摘要和引用 ID 放入 prompt
4. **分支共享**
   把会话和 KV 组织成树，父分支共享，子分支只保存增量；必要时做 Copy-on-Write
5. **冷热分层**
   热 KV 在 GPU，冷分支或低优先级任务迁移到 CPU 或外存
6. **代价感知淘汰**
   不是只看 LRU，还看重算代价、分支价值和任务优先级

#### 四层：指标

必须明确怎么评估：

- 峰值显存
- 平均时延 / P95
- TTFT（首 token 延迟）
- TPOT（每输出 token 时间）
- 任务成功率
- cache hit rate / prefix hit rate
- 重算开销

这套答法非常适合比赛答辩、面试系统设计和简历项目讲解。

## 5. 高频追问：面试官可能怎么继续压你

### 5.1 为什么不把所有历史都摘要掉

因为摘要会损失细节，影响任务成功率。  
正确做法通常不是“全摘要”，而是：

- 最近窗口保留原文
- 远期历史压缩成摘要
- 关键事实单独做结构化记忆

### 5.2 为什么工具结果不能全部丢进 prompt

因为很多工具返回值是“机器可读、对人类大而杂、对模型未必必要”的。  
例如长 JSON、表格、日志、网页正文，直接注入会：

- 拉长 prefill
- 膨胀 KV Cache
- 引入噪声

更合理的是：

- 先筛字段
- 再摘要
- 保留引用
- 需要时按需回查原始数据

### 5.3 Prefix cache 命中高是不是一定好

不一定。  
如果为了提高命中率而引入复杂哈希、维护成本或者错误共享，收益可能被抵消。  
另外，不同工作负载对前缀复用友好度差异很大。

### 5.4 为什么很多优化都要做 benchmark，而不能只看理论

因为推理系统的瓶颈很容易转移：

- 省了显存，可能增加重算
- 提高了吞吐，可能拉高 P95
- 量化后算快了，可能精度掉了
- 做了 PD 分离，可能被网络拖慢

所以面试时要体现出你知道：

> 优化必须在统一硬件、统一 workload、统一指标口径下评估。

## 6. 真正准备面试时，你至少要能讲清楚的 10 个问题

1. `prefill` 和 `decode` 的区别是什么？
2. `KV Cache` 为什么能加速生成？为什么会爆显存？
3. `GQA/MQA` 为什么能省缓存？
4. `vLLM` 和普通 Hugging Face 推理相比核心优势是什么？
5. `PagedAttention` 的设计思想是什么？
6. 什么是 `continuous batching`？它的收益和代价分别是什么？
7. `prefix caching` 适合哪些场景？为什么 Agent 很适合？
8. 权重量化和 KV Cache 优化有什么本质区别？
9. 如果工具调用返回超长 JSON，你怎么避免上下文爆炸？
10. 如果一个 Agent 会分叉探索多个方案，你怎么做缓存共享和内存回收？

如果这 10 个问题你都能讲顺，已经比很多只会调用 API 的候选人更接近相关岗位要求。

## 7. 怎么把这个比赛项目讲成简历项目

建议你以后不要写成“做了一个大模型项目”。要写成更像基础设施项目的说法。

一个更像样的表述是：

> 基于 `vLLM` 搭建本地开源模型推理服务，设计面向智能体长生命周期任务的内存管理方案；围绕 `KV Cache`、前缀复用、工具输出外置和分支共享构建 benchmark，在统一硬件环境下评估显存峰值、时延和任务成功率，并完成可复现优化验证。

如果后面结果做出来，可以继续压缩成 3 个简历点：

- 基于 `vLLM` 实现本地 OpenAI-compatible 推理服务与最小 Agent 工作流
- 设计智能体多轮/工具调用/分支决策 benchmark，完成显存与时延 profiling
- 实现 prompt 去重、工具结果外置、前缀复用或分层缓存优化，在基本不降低成功率前提下优化资源占用

## 8. 你现在最值得优先学的顺序

如果你是初学者，建议按下面顺序准备面试和项目，不要反过来。

1. **先懂角色**
   模型权重、推理、推理引擎、API 服务分别是什么。
2. **再懂请求路径**
   tokenize -> prefill -> decode -> KV Cache 增长。
3. **再懂 vLLM 的几个关键机制**
   `PagedAttention`、`continuous batching`、`prefix caching`、KV offload。
4. **再做最小 agent**
   用户问题 -> 工具调用 -> 观察 -> 最终回答。
5. **最后做内存优化**
   先从上下文压缩和工具结果外置做，再做更深的缓存共享。
